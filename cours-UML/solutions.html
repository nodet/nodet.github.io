1. Identification des acteurs

Le système étudié est le programme qui permet aux personnels de la bibliothèque universitaire d’enregistrer l’emprunt de documents par les étudiants. Ce programme tourne sur un ordinateur muni d’une douchette pour scanner les documents, d’un écran, et d’une carte réseau pour communiquer avec les systèmes informatiques distants.

Indiquez, pour chaque entité, s’il s’agit ou non d’un acteur dans la description du cas d’utilisation "Emprunt d’un document".

  1. Le bibliothécaire qui enregistre l’emprunt

Oui, c’est un acteur : c’est lui qui utilise le système pour obtenir un but, enregistrer l’emprunt du document par l’étudiant.

  1. L’étudiant qui emprunte le document

Ce n’est pas un acteur car il n’utilise pas (directement) le système.

  1. L’ordinateur qui fait tourner le programme

Non, l’ordinateur qui fait tourner le programme n’est pas un acteur. L’ordinateur est necessaire pour que le programme s’execute, comme du courant electrique est nécessaire. Mais le programme n’échange pas de message avec l’ordinateur.

  1. La douchette branchée à l’ordinateur pour lire les codes-barre

Oui, la douchette est un acteur, car le programme va recevoir d’elle les informations concernant les code-barres des documents.

  1. La base de donnée distante qui stocke les documents empruntés par les étudiants

Oui, la BdD distante est un acteur : le programme va échanger des messages avec elle pour stocker et retrouver des informations.

  1. Le système informatique administratif de l’université

Oui, pour la même raison. Par exemple, pour vérifier le numéro d’un étudiant.

2. Identification des acteurs et de cas simples

Le système considéré est une station service de distribution de carburant. Les clients se servent en carburant, et le pompiste remplit les cuves.

  1. Le client se sert de la façon suivante : il prend un pistolet accroché à une pompe et appuie sur la gachette pour faire couler le carburant. Qui est l’acteur dans ce cas d’utilisation ? Le client, le pistolet ou la pompe ?

C’est le client. Un acteur est toujours extérieur au système étudié.

  1. Pierre est pompiste. De temps en temps, il vient à la station pour remplir le réservoir de sa voiture. Pour modéliser cette activité, faut-il un nouvel acteur ?

Non. On considère le rôle de chaque intervenant, et non pas son identité. Le fait que Pierre soit pompiste ne change rien à ce qui se passe lorsqu’il se sert en carburant. Donc il n’y a pas de nouveau rôle.

  1. Certains pompistes sont qualifiés pour effectuer des opérations que tous les pompistes ne peuvent pas faire, comme des opérations de maintenance. Ils sont donc réparateurs en plus d’être pompistes. Comment modéliser cela ?

Le Technicien est une spécialisation du Pompiste, puisqu’il sait faire tout ce que sait faire le pompiste, plus autre chose.

4.1. Cas d’utilisation : identification d’un utilisateur

Liste des acteurs

Préconditions

  • Le bibliothécaire est identifié

Scénario nominal

  • Le bibliothécaire scanne la carte de l’utilisateur à l’aide d’une douchette.

  • Le logiciel présente au bibliothécaire les informations connues sur le porteur de la carte, et un signal sonore de type 'validation' est émis.

  • Ces informations sont : nom et prénom, adresse, liste des documents empruntés.

  • La liste des documents comprend les informations suivantes pour chaque document : type (livre, CD, ou DVD), titre, date de retour maxi.

  • Si la date de retour maxi est égale ou antérieure à la date du jour, le document apparaît avec un marquage spécifique de type 'interdiction'.

  • Si la date de retour maxi est postérieure à la date du jour de 7 jours ou moins, le document apparaît avec un marquage spécifique de type 'attention'.

Enchainement d’erreur

  • Si la carte est inconnue ou illisible, un message est affiché et un signal sonore est émis.

Post-condition

  • Le système est dans l’état "Utilisateur identifié"

4.3. Cas d’utilisation : emprunt d’un document

Liste des acteurs

  • Le bibliothécaire

Préconditions

  • Un utilisateur est identifié

Enchaînement nominal

  • Scan d’un document se termine sans erreur

  • La liste des documents empruntés déjà présente sur l’écran est complétée du document scanné.

Enchaînement d’erreur

  • Si le document est marqué "emprunté", un signal sonore et visuel est émis, un message "Document déjà emprunté" est affiché.

  • Si le document est marqué "réservé" et que l’utilisateur n’est pas celui pour lequel le document est réservé, un signal sonore et visuel est émis, un message "Document réservé" est affiché.

  • Si le nombre de documents déjà empruntés par l’utilisateur est supérieur ou égal à la limite, un signal sonore et visuel est émis, et un message "Trop de document empruntés" est affiché.

Post-conditions

  • Le document est marqué "emprunté" dans la base de donnée du système avec une référence vers l’utilisateur.

Exigences non fonctionnelles

  • Tous les affichages du système seront étudiés pour être lisibles en niveaux de gris aussi bien qu’en couleur.

7. Types de relation

Pour chaque exemple ci-dessous, indiquez si la relation présentée est une généralisation, une agrégation ou une association.

  1. Un pays a une capitale

Déterminons le type de cette relation entre la classe Pays et la classe Ville. Ce n’est pas une généralisation, car un pays n’est pas une ville, et on ne peut pas dire non plus qu’une ville est un pays.

Ce n’est pas une composition car la ville peut exister sans le pays. Par exemple, si le pays est annexé et disparait, et que donc il n’a plus de capitale, la ville est — souhaitons-le — toujours là.

Enfin, la relation a pour capitale ne sera pas non plus une aggrégation. En effet, le fait qu’une ville fasse partie intégrante d’un pays est vrai pour toutes les villes, pas seulement la capitale. On aurait donc une aggrégation entre Pays et Ville pour indiquer à quel pays appartient une ville, et une association simple pour désigner sa capitale.

  1. Une transaction boursière est un achat ou une vente

Les mots est un suggèrent très fortement une relation de type généralisation. Et c’est bien de cela qu’il s’agit.

En effet, il n’est pas possible de transformer une transaction boursière, qui est nécessairement soit un achat, soit une vente, et ne peut passer d’un état à un autre. D’autre part, il existe bien des éléments communs aux achats et aux ventes (la date, l’identité de l’acheteur ou du vendeur, etc.) qui seront donc portés par une classe commune Transaction.

  1. Les fichiers contiennent des enregistrements

Un enregistrement n’est pas un type de fichier, comme une ligne n’est pas un type de table dans une base de donnée, et une fiche n’est pas un type de classeur dans votre collection de cours.

Comme la ligne d’une table, et contrairement à une fiche de cours, un enregistrement n’a pas d’existence propre : si on enlève un enregistrement d’un fichier, ou une ligne d’une base de donnée, ceux-ci disparaissent tout simplement. Il s’agit donc d’une relation d'aggrégation : l’enregistrement est constitutif du fichier, et n’existe pas en dehors du fichier.

  1. Une personne utilise un langage de programmation dans un projet

Un langage de programmation n'est pas un projet, et vice-versa. Donc il ne s’agit pas d’une généralisation. De même, un langage de programmation et un projet ne sont pas inclus l’un dans l’autre, donc il n’y a pas d’aggrégation, et encore moins de composition. Il s’agit donc simplement d’une association.

  1. Les modems et les claviers sont des périphériques d’entrées/sorties

Les modems et les claviers sont des objets qui font tous les deux partie de la classe des périphériques d’entrées/sorties. Ce sont deux sous-types de cette classe, et un objet d’un type ne peut pas être changé en un objet de l’autre type. Nous avons donc ici une généralisation.

11. Python : Vehicles

class Vehicle:

  def __init__(self, name, capacity):
    self.name = name
    self.capacity = capacity

  def fare(self):
    return self.capacity * 100

  def __repr__(self):
    return f"{self.name}: {self.fare()}"


class Car(Vehicle):

  def __init__(self, name):
    super().__init__(name, 5)


class Bus(Vehicle):

  def fare(self):
    return super().fare() * 1.1


v = Car("VW Touran")
print(v)

v = Bus("School Volvo", 50)
print(v)

12. Python : un robot très simple

class Robot:
    
    def __init__(self, x, y):
        self.x = x
        self.y = y
        self.direction = 'Nord'

    def __repr__(self):
        return [self.x, self.y, self.direction].__repr__()
    
    def delta_x(self):
        if self.direction == 'Est':
            return 1
        elif self.direction == 'Ouest':
            return -1
        else:
            return 0

    def delta_y(self):
        if self.direction == 'Nord':
            return 1
        elif self.direction == 'Sud':
            return -1
        else:
            return 0

    def vitesse():
        return 1

    def avance(self):
        self.x += self.delta_x()
        self.y += self.delta_y()

    def droite(self):
        old_direction = self.direction
        if self.direction == 'Nord':
            self.direction = 'Est'
        elif self.direction == 'Est':
            self.direction = 'Sud'
        elif self.direction == 'Sud':
            self.direction = 'Ouest'
        elif self.direction == 'Ouest':
            self.direction = 'Nord'
        assert(old_direction != self.direction)


r = Robot(2,4)
assert(r.x == 2 and r.y == 4 and r.direction == 'Nord')
r.avance()
print(r)
assert(r.x == 2 and r.y == 5)

r.droite()

13. Python : que fait ce programme ?

class Personne:

    def __init__(self):
        self.stock = {'pain': 1, 'ble': 0, 'farine': 0}

    def mange(self):
        if self.stock['pain'] <= 0:
            print("J'ai faim !")
            return
        self.stock['pain'] -= 1
        print("J'ai bien mangé.")

    def souhaite_obtenir(self):
        return [('pain', 1)]

    def donne_a(self, personne):
        assert (self != personne)
        list_of_pairs = personne.souhaite_obtenir()
        for p, q in list_of_pairs:
            q = min(q, self.stock[p])
            if q > 0:
                self.stock[p] -= q
                personne.stock[p] += q

    def transforme(self, ingredient, produit):
        q = self.stock[ingredient]
        if q > 0:
            self.stock[ingredient] -= q
            self.stock[produit] += q

class Paysan(Personne):
    def produit(self):
        self.stock['ble'] += 3

class Meunier(Personne):
    def produit(self):
        self.transforme('ble', 'farine')
    def souhaite_obtenir(self):
        return [('pain', 1), ('ble', 3)]

class Boulanger(Personne):
    def produit(self):
        self.transforme('farine', 'pain')
    def souhaite_obtenir(self):
        return [('farine', 3)]

personnes = [Paysan(), Meunier(), Boulanger()]
for i in range(4):
    for p in personnes:
        p.mange()
    for p in personnes:
        p.produit()
    for p in personnes:
        for p_prime in personnes:
            if p != p_prime:
                p.donne_a(p_prime)
    print()

for p in personnes:
    print(p.stock)

14. Python : les Mangenombres

class MangeNombres:

  def __init__(self, p):
    self.p = p                           (1)
    print(p)                             (2)
    self.suivant = None                  (3)

  def mange(self, q):                    (4)
    if (q % self.p == 0):
      return                             (5)
    if self.suivant:
      self.suivant.mange(q)              (6)
    else:
      self.suivant = MangeNombres(q)     (7)

N = 2
m = MangeNombres(N)
for i in range(N+1, 20):
  m.mange(i)

  1. Tous les multiples de p seront mangés

  2. p étant premier, on l’affiche

  3. On créera le MangeNombres suivant quand on nous donnera un nombre non multiple de p à manger

  4. La fonction qui 'mange' le nombre ou le passe au suivant

  5. Si q est un multiple de p, on le mange et c’est fini pour lui

  6. Sinon on le donne au MangeNombres suivant s’il existe

  7. q n’a été mangé ni par ce MangeNombres, ni par aucun de ses prédécesseurs. Donc q est premier, et créer le MangeNombres suivant va (1) afficher q et (2) s’assurer que ses multiples seront mangés

15. Python : A budget app

class Category:
  """
  Records a budget, a list of expenses.
  Budget can be updated later
  """

  def __init__(self, name, remaining):
    self.name = name
    self.expenses = []
    self.remaining = remaining

  def __repr__(self):
    return (self.name + '('
      + str(self.remaining) + ' -- '
      + str(self.sum_of_expenses()) + ')')

  def add_expense(self, amount):
    self.expenses.append(amount)
    self.remaining -= amount

  def sum_of_expenses(self):
    return sum(self.expenses)

  def add_to_budget(self, amount):
    self.remaining += amount


class Budget:
  """
  This class allows to record expenses and
  categorize them into various types of
  expenses
  """

  def __init__(self, total):
    self.initial_budget = total
    self.total = total
    self.categories = {
      # Each category will be a name and a list
      # of amounts
    }

  def sum_of_expenses(self):
    somme = 0
    for name, category in self.categories.items():
      somme += category.sum_of_expenses()
    return somme

  def print_status(self):
    print(f'Initial budget: {self.initial_budget}')
    print(f'Remaining budget: {self.total}')
    somme = self.sum_of_expenses()
    print(f'Sum of expenses: {somme}')
    for key, val in self.categories.items():
      print(val)
    print()

  def add_category(self, name, amount):
    assert(name not in self.categories)
    self.categories[name] = Category(name, amount)

  def add_expense(self, cat, amount):
    assert(cat in self.categories)
    self.categories[cat].add_expense(amount)
    self.total -= amount
    assert(self.total + self.sum_of_expenses() == self.initial_budget)

  def balance(self, source_category, destination_category, amount):
    assert(source_category in self.categories)
    assert(destination_category in self.categories)
    self.categories[source_category].add_to_budget(-amount)
    self.categories[destination_category].add_to_budget(amount)

  def proportions(self):
    total = self.sum_of_expenses()
    l = []
    for f, c in self.categories.items():
      this_cat = c.sum_of_expenses()
      l.append((f, this_cat / total))
    return l

b = Budget(100)
b.print_status()
b.add_category('Food', 20)
b.add_category('Transport', 10)
b.print_status()
b.add_expense('Food', 1.4)
b.add_expense('Transport', 4.6)
b.print_status()
b.balance('Food', 'Transport', 2.8)
b.print_status()
print(b.proportions())
